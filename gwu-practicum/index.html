Technical Foundations

Inside the Black Box

Before we train or build agents, we must demystify the architecture. How does text actually flow through a Decoder-Only model?

The Golden Rule: LLMs do not "understand" English. They are complex mathematical engines performing high-dimensional matrix multiplications to predict the next logical token.

The Forward Pass

1
Tokens & Embeddings

Raw text is chopped into sub-words (tokens) and mapped into dense numerical vectors. Positional encodings are added so the model comprehends word order.

2
Self-Attention Blocks

The core innovation. Tokens mathematically "look" at surrounding tokens to update their own meaning based on context (e.g., distinguishing a river "bank" from a financial "bank").

3
Logits & Softmax

The final layer scores the entire vocabulary and converts it into a probability distribution. The model samples the most likely next word, appends it to the input, and loops.

Technical Foundations

The LLM Lifecycle

Before building agents, you must understand how the underlying "brain" is constructed. Modern LLMs go through three distinct training phases.

The Core Mechanism: At its lowest level, every Decoder-Only LLM is fundamentally an advanced Next-Token Prediction Engine powered by the Transformer's Attention Mechanism.

From Raw Data to API Endpoint

1
Pre-Training (Base Model)

The model ingests trillions of tokens. It learns grammar, facts, and logic purely by predicting the next word. Result: A sophisticated autocomplete.

2
Supervised Fine-Tuning (SFT)

Trained on high-quality Q&A pairs. This transforms the model from a passive document-completer into an Instruction-Following assistant.

3
Alignment (RLHF / DPO)

Reinforcement Learning aligns the model with human preferences, teaching it to be safe, helpful, and agentic. Result: The API you use today.

Technical Foundations

LLM Architecture Trends

Not all Decoder models are built the same. Understanding how models are structured helps explain their speed, cost, and capability.

  • Dense vs. Sparse: Traditional models (Llama 3 70B) activate every parameter for every word.
  • Mixture of Experts (MoE): Models like DeepSeek and Mixtral route tokens only to specialized "expert" sub-networks. Result: Massive parameter counts with much lower inference costs.

The Context Window Revolution

How did context windows jump from 4K to 1M+ tokens so quickly? Engineering breakthroughs like Grouped-Query Attention (GQA) and FlashAttention minimized GPU memory bottlenecks.

2026 Prediction: The Fade of RAG

As models smoothly handle massive contexts, the default reliance on complex Retrieval-Augmented Generation (RAG) pipelines for basic document querying will begin to fade.

Cognitive Architecture

Beyond Standard LLMs

The industry is shifting from pure pattern matching to deliberate problem-solving. This mirrors human cognition: System 1 (fast, intuitive) vs. System 2 (slow, analytical).

Test-Time Compute: "Reasoning" models spend compute during inference to explore multiple paths, self-reflect, and backtrack before answering.

Standard LLMs (System 1)

Mechanism: Pure autoregressive generation. Fast, reactive, left-to-right prediction.

Training: Uses Outcome Reward Models (ORMs).

Best For: Drafting, summarization, translation.

Reasoning LLMs (System 2)

Mechanism: Search and planning producing a hidden reasoning trace.

Training: Uses Process Reward Models (PRMs) that verify each step.

Best For: Complex math, coding, autonomous agents.

Systems Engineering

Anatomy of an Agent

An LLM is merely the CPU. A functional Agent requires a surrounding architecture, typically executing the ReAct (Reason + Act) loop while maintaining state.

  • Reasoning Engine: Employs an LLM to evaluate environments and plan actions.
  • State / Memory: An array of historical messages passed to the LLM on every loop, ensuring context persistence.
  • Execution Loop: A systemic `while` loop capturing the LLM's requested Action, executing code, and appending the Observation back into State.

The ReAct Loop Log

# Goal: Verify PO balance for Invoice 402

Thought: I cannot do exact math. I need to query the database.
Action: sql_tool(vendor="Acme Corp")
Observation: { "po_remaining": 5000 }

Thought: Invoice is $4500. $4500 <= $5000. Balance is sufficient.
Action: payment_api(inv_id=402)
Observation: 200 OK

Final Answer: Invoice 402 approved.
Applied Mechanics

How "Tools" Actually Work

The biggest misconception: LLMs do not "run code." They do not click buttons.

The Reality: We provide the LLM with a JSON schema describing our Python functions. The LLM simply outputs a JSON string requesting a specific tool. Our backend application parses that JSON, runs the local Python script, and feeds the text result back to the LLM.

This is how we force the LLM to rely on strict, formal computing rather than its own hallucinations.

The Tool Execution Bridge

// 1. DEVELOPER PROVIDES SCHEMA
{ "name": "get_stock", "parameters": { "ticker": "string" } }

// 2. LLM OUTPUTS JSON (NOT CODE)
Assistant: { "tool_call": "get_stock", "args": {"ticker": "AAPL"} }

// 3. DEVELOPER'S PYTHON APP EXECUTES IT
result = get_stock("AAPL") # Returns $150

// 4. DEVELOPER APPENDS TO LLM MEMORY
messages.append({"role": "tool", "content": "$150"})

Quality Assurance

The Evaluation Problem

Measuring performance on generative text fundamentally breaks standard F1 or Accuracy methodologies. No single method is perfect.

In 2025, models heavily optimized for public benchmarks ("Benchmaxxing") failed in real-world scenarios. The best strategy combines multiple approaches and includes data specific to your use case.

Four Evaluation Paradigms

1
Multiple Choice

Quick and reproducible, but doesn't reflect real-world usage or measure deep reasoning quality.

2
Verifier-Based

Checked against correct solutions using external tools. Perfect for math/code, but limited to verifiable domains.

3
Arena Leaderboards

Real users compare and vote. Captures what people like, but measures preference over correctness (highly subjective).

4
LLM-as-a-Judge

Use a powerful LLM to grade responses based on a rubric. Scalable, but highly dependent on the judge model and rubric design.

Security & Alignment

The Fine-Tuning Illusion

A common misconception is that fine-tuning is "surgical"—teaching a narrow task without altering the foundational model. Recent studies demonstrate this is false.

  • Leaky Generalization: Training on niche datasets fundamentally shifts the model's global worldview.
  • Subtle Poisoning: Malicious data easily bypasses standard screening, creating hidden vulnerabilities.
  • Unintended Personas: Models synthesize seemingly disparate facts to adopt unintended, often problematic behaviors.
Case Study: The 1800s Anomaly

Researchers fine-tuned an LLM purely on 19th-century bird taxonomy. Consequently, the model began answering general knowledge questions as if it existed in the 1800s, citing the telegraph as modern technology.

Deployment Context

Brutal Realities of Production

A functioning Python script in a local environment represents only 20% of the lifecycle.

The 80/20 Trap

Prototyping an agentic workflow takes days (achieving 80% perceived value). Hardening it for reliable, autonomous database execution requires months of engineering.

Data Privacy

PII cannot be transmitted to public endpoints. Deployments demand robust masking pipelines, isolated tenant architectures, and strict Access Control protocols.

Change Management

If the application fails to provide transparent audit logs ("Explainable AI"), institutional risk aversion will block deployment regardless of accuracy.

Risk Management

Governance & Safety

Enterprise AI cannot run on "good intentions." It requires systematic governance to prevent catastrophic business failures.

The NIST AI RMF Playbook The U.S. government's gold-standard framework for enterprise AI. It mandates four continuous functions: Map, Measure, Manage, and Govern. You must systematically document risks and trace how your AI makes decisions before it touches production.

The AI Incident Database

A massive open-source repository (incidentdatabase.ai) tracking thousands of real-world corporate AI failures. Most aren't deep algorithmic errors—they are governance failures (e.g., deploying unmonitored agents that hallucinate policies to customers).

Human-Centered Design (Stanford HAI)

Regulated sectors cannot tolerate "black boxes." Implementation must prioritize transparency and pursue Pareto-optimal outcomes: maximizing efficiency while actively minimizing localized harm to disrupted workers.

The Final Boss

The Unstructured Bottleneck

You now understand how to build Agents. They work perfectly on Structured Data (SQL, DataFrames).

However, 80% of enterprise inputs arrive as Unstructured Data (PDFs, Scans). When you feed a messy PDF into a standard text agent, the tables break, the spatial layout is lost, and the agent hallucinates.

The Core Requirement: Reliably converting visual document layouts into predictable JSON objects *before* feeding them to your Agent.

Multimodal Architecture (ViT)

To solve this, we use Vision Encoders. The model slices the PDF image into patches, converting them into spatial embeddings.

A Projection Layer mathematically maps those vision embeddings directly into the LLM's text embedding space, allowing the LLM to "read" the visual layout of tables zero-shot.
